# AGENTS.md

## Project Overview

**nozero** is an AI-powered scheduling application built with Next.js 16 (App Router), React 19, and Supabase as the backend. It features Google Calendar bi-directional sync, natural-language event creation via AI chat, conflict detection, analytics, and event invitations via email.

## Tech Stack

- **Runtime / Package Manager:** Bun (pinned to 1.3.11 via `packageManager` in `package.json`)
- **Framework:** Next.js 16 with App Router and Turbopack
- **Language:** TypeScript 6 (strict mode)
- **UI:** React 19, Tailwind CSS v4, shadcn/ui components (Radix UI primitives), Framer Motion
- **Backend:** Supabase (Postgres, Auth, Realtime) ‚Äî `nozero` schema with RLS
- **Auth:** Supabase Auth, Google provider (configured in Supabase dashboard)
- **AI:** Vercel AI SDK with OpenRouter provider (`@ai-sdk/openai`, `@openrouter/ai-sdk-provider`)
- **Email:** Resend
- **Forms:** react-hook-form + Zod validation
- **Analytics:** @vercel/analytics

## Linting and Formatting

This project uses **Biome** via **Ultracite** for both linting and formatting.

- Config: `biome.jsonc` extends `ultracite/biome/core`, `ultracite/biome/react`, `ultracite/biome/next`
- The `convex/_generated/` directory is excluded from linting
- Always run `bun run lint` before committing to verify your changes pass
- Run `bun run lint:fix` to auto-fix issues

Several lint categories have rules explicitly relaxed in `biome.jsonc` (a11y, complexity, correctness, performance, style, suspicious, security). Follow the existing patterns rather than re-enabling stricter rules.

## TypeScript

## Code Conventions

- **Imports:** Use `@/` path aliases (e.g., `@/components/ui/button`, `@/lib/utils`)
- **Components:** Follow shadcn/ui patterns ‚Äî components in `components/ui/` are Radix-based primitives styled with Tailwind and `class-variance-authority`
- **Icons:** Use `@hugeicons/react` (configured as the icon library in `components.json`)
- **Styling:** Tailwind CSS v4 utility classes with CSS variables for theming; use `cn()` from `@/lib/utils` for conditional class merging (clsx + tailwind-merge)
- **Forms:** Use react-hook-form with Zod schemas for validation
- **State:** Server state via Supabase server client in API routes / RSC; local React state for UI state. Supabase Realtime subscriptions are available for `nozero.{profiles,events,categories,invitations}` via hooks in `hooks/use-realtime-*.ts` (see below).
- **Server vs Client:** Next.js App Router conventions ‚Äî components are server components by default; add `"use client"` directive only when needed

## Realtime

User-scoped row subscriptions for any of the four `nozero` tables:

```tsx
"use client";
import { useState } from "react";
import { useRealtimeUserEvents } from "@/hooks/use-realtime-nozero";

export function EventsList() {
  const [tick, setTick] = useState(0);
  useRealtimeUserEvents(() => setTick((n) => n + 1)); // bump on INSERT/UPDATE/DELETE
  // re-fetch (TanStack Query, SWR, or plain useEffect) keyed on `tick`...
}
```

`hooks/use-realtime-table.ts` is the generic primitive; `hooks/use-realtime-nozero.ts` adds session-scoped filters (`user_id=eq.<auth.uid()>`, `organizer_user_id=eq.<auth.uid()>`, etc.). The filter pushes work to the server ‚Äî clients only get rows they're entitled to under RLS.

## Supabase Backend

- Schema `nozero` with four tables: `profiles`, `events`, `categories`, `invitations`.
- DDL lives in `supabase/migrations/*.sql`; apply with `psql "$SUPABASE_DB_URL" -f ‚Ä¶`.
- RLS is on for all tables; policies key on `auth.uid()` / `auth.jwt() ->> 'email'`.
- Public invite landing reads go through `SECURITY DEFINER` RPC `nozero.invitation_by_token(text)` ‚Äî never SELECT directly from anon.
- Realtime publication `supabase_realtime` includes all four tables.
- Generated types: `bun run types:gen` writes `types/database.ts`.
- Sign-in callback is `app/auth/callback/route.ts` ‚Äî exchanges code for session and upserts Google `provider_token` / `provider_refresh_token` into `nozero.profiles`.

## Learned User Preferences

- Primary nozero login (email+password or OAuth) is separate from linked mail/calendar identities; one session may attach many Google, CalDAV, and IMAP accounts‚Äîeach identity kept completely separate by `account_email`, never merged across logins or identities.
- Per-account calendar subscription is configured in Settings, not by auto-importing every shared/org calendar.
- Sidebar calendar visibility toggles must persist across sessions; only subscribed calendars sync events.
- Settings uses three sections: Preferences (login identity + global prefs), Accounts (connected email/calendar identities), Connections (Krisp, GitHub context). Primary login must not share a form with connected accounts; account management stays out of the calendar sidebar.
- Calendar labels should show user-edited friendly account names (e.g. `julian.tedstone@coherence.digital` ‚Üí "Julian, Coherence" or "Jules"), not generic "Calendar"/"Primary" and not auto-re-derived from email on save.
- CalDAV credential edits must retain stored passwords without forcing re-entry on Save/Reconnect.
- Calendar and email must be offline-first: instant reads from a local mirror; sync never blocks initial paint; sync is manual refresh or background only; the app must work fully offline with data as of the last sync. Email inbox sync uses direct Gmail API and IMAP (not Soma/Anansi); sync errors must show in the UI, not as silent empty inboxes.
- Context, email, and board are first-class top-level tabs alongside calendar; all three-column views share one shell: default 25% / 50% / 25%, draggable to max 33% each, left/right panels toggle show/hide (widths persist in localStorage). Nested sidebar blocks use a divider keyline, compact title, and expand/collapse (contracted by default‚Äîemail Accounts and Search). Meeting context detail zooms in from Context/calendar/HUD. Context tab: left stream nav; center markdown/csv/yaml editor with validated save; right stream shelf with file tree (add/open files) plus toggles for summary, Updates, CRM, Tickets, Repo actions, Messages, Events. Email: horizontal nav, white "+ New Message", Unread¬∑Tracking¬∑All, search/stream filters, paginated inbox (~20), row hover actions (reply/forward/archive/track), centre thread + bottom compose bar, avatar/settings bottom-left. Board (GitHub #17): Group by Stream/Owner/Approver and Sort by ticket no / next action (desc) / priority (asc); stream swimlanes with per-row show/hide; subject-only cards; "+ New task" left of search; detail shelf overlays up to 50%; card field order Owner ‚Üí Approver/Approval ‚Üí Stream/Priority ‚Üí Next Action/Recurrence; collapsed Recurring/Completed/Scheduled edge columns; no Flightdeck branding or helper copy. Panel headers omit visible header/body keylines. Avoid placeholder copy in empty states. Calendar chrome uses month/week/day tabs.
- Event UI: participant fields need Soma/message-history contact autocomplete; notes render plain text; time picker uses 5-minute increments; event pills use subscribed calendar colors on flat/2D panels without left accent borders.
- Event detail: group fields into toggleable What / Where / When sections (default order What ‚Üí Where ‚Üí When, globally configurable in settings) plus a bottom Who section (participants, Assign/project-code dropdown, Flightdeck stream dropdown, calendar-account switcher). Replace the Edit Event header with a HuD (title, organizer status, friendly datetime, map/video launchers, countdown). Lock non-owned meetings (padlock + organizer tooltip) with Google-style edit scope: no time/title/remove-attendee edits, but add-participant when the provider allows. Extract Teams/Zoom/Meet/Slack links from invite body into the meeting URL field. Assign and stream pickers are single aligned dropdowns in What.
- Recurring meetings should follow Google Calendar recurrence rules and edit scope (this/ following / all).
- Optional meeting Assign uses per-account account codes for time-sheeting; codes are CRUD-managed in Settings, upsertable from the event form, and archive-only (never deleted).

## Learned Workspace Facts

- Google and CalDAV tokens/credentials live in `nozero.profiles.preferences` (`connectedTokens`, `connectedCalDav`, `connectedAccounts` metadata); IMAP in `connectedImap` plus `nozero.imap_credentials` via `lib/imap-credentials.ts`; registry can desync from tokens when `PUT /api/accounts` full-replaces metadata or unscoped `localStorage` races occur. Events stay bound to their source linked account (`accountEmail`); cross-account moves only via explicit Who-section account switcher, never silent migration on save.
- CalDAV connect/sync (incl. iCloud preset at `https://caldav.icloud.com`, app-specific password): `POST /api/accounts/caldav/connect`, `lib/caldav-sync.ts` with `tsdav` `DAVClient`; stored as `type: "caldav"`.
- Multi-account Google sync pulls linked accounts via `pullAllGoogleCalendarAccounts()` in `lib/google-accounts-sync.ts`; password-login profiles must keep `profiles.provider=email`‚Äîsynthetic `primary-google` sidebar calendar is for Google sign-in only, or synced events under linked identities are hidden.
- Calendar subscriptions and visibility filtering: `lib/calendar-subscriptions.ts`, `lib/calendar-subscription-utils.ts` (`eventMatchesVisibleSubscriptions`); visibility API at `/api/calendar/visibility`.
- Windowed sync: initial pull ¬±3 months (`lib/sync-window.ts`, `lib/calendar-sync-range.ts`); background extension via `POST /api/calendar/sync/extend` in 3-month chunks.
- Contact autocomplete: `/api/contacts/suggest` (Soma client in `lib/soma-client.ts`) plus local invitation/event attendee fallbacks.
- Local mirror: IndexedDB `nozero-local-mirror` in `lib/local-mirror/db.ts`; calendar/email UI reads mirror first; background sync in `hooks/use-background-sync.ts`; hydrate via `calendar-hydrate` / `email-hydrate`; email threads list defaults to `sync=false`.
- Settings form saves must merge `preferences` blobs so OAuth/calendar token fields are not wiped when updating appearance/time prefs. Per-account GitHub context-message repo bindings and email account visibility/accounts-expanded UI state persist the same way via `lib/email-preferences.ts` (server-side per user, not local-only).
- Account codes: `nozero.account_codes` table plus `/api/account-codes` and `lib/account-codes.ts`; scoped to linked account email; archived codes stay out of pickers but are never hard-deleted.
- Production (zero.nopilot.co) on Jupiter uses Supabase **gily** (`gilyyzjsasyhrwterjor.supabase.co`); PostgREST must expose schema `nozero`.
- Jupiter Lane A deploy: secrets via `op inject` + `OP_SERVICE_ACCOUNT_TOKEN` from `/root/npt-core/.env.op` (service account, not `op signin`); `deploy/lib/host-deploy.sh` aborts build on unresolved `op://` and chmod `600` on injected `.env.*.local`; runtime reads injected env only. All `NEXT_PUBLIC_*` must be real values before `bun run build` ‚Äî literal `op://` in the client bundle breaks Supabase auth ("Invalid email or password"). `DEFAULT_SUPER_ADMIN*` is ops/1Password parity for manual admin provisioning only.
- Context, email, and board UI: shared `components/three-column-layout.tsx` + `components/collapsible-sidebar-section.tsx`; Context tab in `components/context-view.tsx` with meeting drill-down in `components/meeting-context-detail.tsx` and bundle API `POST /api/context/meeting` (ctx gateway, Tower, Krisp; CRM via `lib/hydration-client.ts`); email in `components/email-view.tsx` with prefs in `lib/email-preferences.ts`, inbox sync `POST /api/email/sync` / `lib/mail-sync.ts`, agent drafts `POST /api/email/draft-reply`; linked Google mail/calendar OAuth at `/api/auth/google/connect` ‚Üí `/api/auth/google/callback` (redirect URI per host in Google Cloud Console, separate from Supabase login); Flightdeck board in `components/flightdeck-board-view.tsx` + `/api/flightdeck/board` (`lib/flightdeck-client.ts`).
